### 4.2 P0 功能详细设计

#### 功能 1: 智能注入 (Smart Injection)

```python
class SmartInjector:
    """智能记忆注入器"""
    
    def inject(self, user_query: str, context: Dict) -> List[Memory]:
        # 1. 意图分类
        intent = self.classify_intent(user_query)
        
        # 2. 动态调整注入量
        if intent in ["simple_question", "greeting"]:
            top_k = 3  # 简单问题，少注入
        elif intent in ["code_generation", "analysis"]:
            top_k = 10  # 复杂任务，多注入
        else:
            top_k = 5  # 默认
        
        # 3. 相关性过滤
        memories = self.memory_manager.retrieve(
            query=user_query,
            top_k=top_k * 2,  # 多检索一些
            filters={"relevance": "high"}  # 只取高相关
        )
        
        # 4. 去重和精简
        memories = self.deduplicate(memories)
        memories = self.compress(memories, target_tokens=5000)  # 目标 5K tokens
        
        return memories[:top_k]
```

#### 功能 2: 记忆压缩与摘要

```python
class MemoryCompressor:
    """记忆压缩器"""
    
    def compress(self, memories: List[Memory]) -> CompressedMemory:
        """将多条记忆压缩为一条摘要"""
```

## 五、结论与建议

### 5.1 核心问题回答

**问题 1: claw-mem 是否解决了 OpenClaw 记忆系统的所有核心问题？**

```
答案: 否，还有重要欠缺

已解决:
✅ 记忆存储和持久化
✅ 注意力管理（检索）
✅ 上下文管理（注入）
✅ 性能优化（极致性能）
✅ Local-First 架构
✅ CJK 支持

欠缺:
❌ Token 消耗问题 ⭐ 最重要
❌ 语义理解缺失
⚠️ 自动压缩不足
⚠️ 记忆关联缺失
```

**问题 2: Token 消耗是技术问题还是业务问题？**

```
答案: 两者都是，需要分层解决

技术问题:
- 记忆注入策略过于简单
- 没有压缩和摘要机制
- 没有分层注入

业务问题:
- Token 消耗直接影响成本
- 用户对成本敏感
- 需要在效果和成本间平衡

解决路径:
短期: Memory 层优化（智能注入 + 压缩 + 分层）
中长期: 架构层优化（系统提示 + 对话历史）
长期: 业务层优化（模型选择 + 批量处理）
```

**问题 3: 应该在 Memory 层解决 Token 问题吗？**

```
答案: 短期可以，长期不够

短期 (Memory 层):
✅ 最容易优化
✅ 快速验证效果
✅ 用户立即受益
✅ 不影响核心架构
预计节省: 30-70% Memory Token

长期 (架构层):
⚠️ Memory 层只占总 Token 的 30-50%
⚠️ 需要系统提示优化
⚠️ 需要对话历史管理
预计节省: 50-80% 总 Token
```
